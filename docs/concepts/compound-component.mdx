# Compound Component 패턴

## API reference

- `createContext<T | undefined>(undefined)`: Root 밖에서 하위 컴포넌트를 쓴 상황을 구분할 수 있게 기본값을 `undefined`로 둔다.
- `Provider value`: Root가 소유한 공유 상태와 상태 변경 함수를 하위 컴포넌트에 내려준다.
- `useContext` guard: 하위 컴포넌트는 전용 hook으로만 Context를 읽고, Provider가 없으면 바로 에러를 던진다.
- subcomponent 조합: `Tabs.Root`, `Tabs.Trigger`처럼 한 기능의 부품을 묶어 노출하고, 배치는 사용하는 쪽이 정한다.

## 쓰는 곳

Compound Component는 Tabs, Accordion, Select, Dropdown처럼 한 UI가 여러 하위 조각으로 나뉘고, 그 조각들이 같은 상태를 공유할 때 쓴다.

Root는 상태의 소유자이고, subcomponent는 Context를 통해 필요한 값만 읽는다. 그래서 사용하는 쪽은 JSX 구조를 자유롭게 조립하면서도 상태 연결 규칙은 한곳에 모을 수 있다.

단순히 부모가 자식에게 props 몇 개를 넘기면 충분한 구조라면 쓰지 않는다. 하위 컴포넌트의 위치와 개수가 화면마다 달라질 때 이 패턴의 값이 생긴다.

## 바로 쓰는 코드

```tsx title="Tabs core"
type TabsContextValue = {
  selectedValue: string;
  selectValue: (nextValue: string) => void;
};
const TabsContext = createContext<TabsContextValue | undefined>(undefined); // [!code highlight]
const useTabs = () => {
  const context = useContext(TabsContext);

  if (context === undefined) {
    throw new Error("Tabs component must be used within Tabs.Root"); // [!code highlight]
  }

  return context;
};
const TabsRoot = ({ children }: { children: ReactNode }) => {
  const [selectedValue, selectValue] = useState("overview");
  const contextValue = useMemo(
    () => ({ selectedValue, selectValue }), // [!code highlight]
    [selectedValue],
  );

  return <TabsContext.Provider value={contextValue}>{children}</TabsContext.Provider>; // [!code highlight]
};

const TabsTrigger = ({ value, children }: { value: string; children: ReactNode }) => {
  const { selectedValue, selectValue } = useTabs();

  return (
    <button type="button" aria-selected={selectedValue === value} onClick={() => selectValue(value)}>
      {children}
    </button>
  );
};

export const Tabs = { Root: TabsRoot, Trigger: TabsTrigger }; // [!code highlight]
```

## 실수 포인트

- Context 기본값을 의미 있는 fallback처럼 채우면 Root 없이 쓴 버그가 늦게 드러난다.
- Provider `value`에 객체를 매 렌더마다 새로 만들면 Context 소비 컴포넌트가 불필요하게 다시 렌더링될 수 있다.
- subcomponent가 서로 props를 직접 전달하기 시작하면 Root가 상태의 단일 소유자라는 장점이 사라진다.
- 조합 API를 만든다는 이유로 전체 구현을 숨기면 안 된다. 실제 상태 기준은 Root에 남겨 둔다.
