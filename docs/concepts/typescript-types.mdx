# TypeScript 타입 설계

## API reference

- type alias: 객체, 함수, primitive 조합에 이름을 붙인다.
- union: 값이 여러 후보 중 하나일 때 가능한 값을 제한한다.
- discriminated union: `status`, `type`, `kind` 같은 tag로 분기마다 필요한 값을 좁힌다.
- generic: 호출자가 넘긴 타입을 반환값, props, callback까지 잃지 않고 이어준다.
- React props type: component가 받는 데이터와 event callback 경계를 component 가까이에 둔다.

## 쓰는 곳

- API 응답, 비동기 상태, component props, 재사용 hook처럼 데이터 경계가 흐려지기 쉬운 곳에 쓴다.
- 상태마다 필요한 값이 다르면 optional field 여러 개보다 discriminated union을 쓴다.
- 화면마다 필요한 필드가 다르면 API 타입 전체를 넘기지 말고 props 타입을 따로 둔다.

## 바로 쓰는 코드

```ts title="async-state.ts"
type AsyncState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; message: string };

const getMessage = (state: AsyncState<{ name: string }>) => {
  if (state.status === "success") return state.data.name;
  if (state.status === "error") return state.message;
  return "대기 중";
};
```

```tsx title="List.tsx"
type ListProps<T> = {
  items: T[];
  renderItem: (item: T) => ReactNode;
};

const List = <T,>({ items, renderItem }: ListProps<T>) => {
  return <ul>{items.map((item) => renderItem(item))}</ul>;
};

const UserList = ({ users }: { users: { id: string; name: string }[] }) => {
  return (
    <List
      items={users}
      renderItem={(user) => <li key={user.id}>{user.name}</li>}
    />
  );
};
```

## 실수 포인트

- `any`로 외부 입력과 component 경계를 지우지 않는다.
- optional 속성 여러 개로 상태 조합을 만들면 impossible state가 생긴다.
- generic은 호출자가 타입을 잃지 않아야 할 때만 쓴다.
- 한 번만 쓰는 짧은 props 타입은 inline으로 둬도 된다.
- 타입을 맞추려고 런타임 구조를 복잡하게 만들지 않는다.
