# Optimistic Update

## API reference

- snapshot: 요청 전 값 중 실패하면 되돌릴 최소 단위다.
- optimistic state: 서버 응답 전에 먼저 화면에 반영하는 임시 state다.
- rollback: 요청 실패 시 snapshot으로 이번 요청의 변경만 되돌린다.
- reconcile: 요청 성공 시 optimistic state를 서버 응답 값으로 확정한다.
- temporary id: 서버 id가 오기 전까지 optimistic state를 찾는 임시 id다.

## 쓰는 곳

좋아요, 저장, 삭제, 댓글 작성처럼 성공 가능성이 높고 즉시 반응이 중요한 action에 쓴다.

서버 응답이 정렬, 권한, id를 바꿀 수 있으면 성공 후 reconcile까지 처리한다.

## 바로 쓰는 코드

```tsx title="CommentForm.tsx"
type Comment = {
  id: string;
  status?: "saving";
  text: string;
};

const CommentForm = ({ initialComments }: { initialComments: Comment[] }) => {
  const [comments, setComments] = useState(initialComments);

  const submitComment = async (text: string) => {
    const tempId = `temp:${crypto.randomUUID()}`;
    const snapshot = { tempId };

    setComments((current) => [
      ...current,
      { id: tempId, status: "saving", text },
    ]); // [!code highlight]

    try {
      const savedComment = await createComment({ clientId: tempId, text });

      setComments((current) =>
        current.map((comment) =>
          comment.id === tempId ? savedComment : comment,
        ),
      ); // [!code highlight]
    } catch {
      setComments((current) =>
        current.filter((comment) => comment.id !== snapshot.tempId),
      ); // [!code highlight]
    }
  };

  return <CommentList comments={comments} onSubmit={submitComment} />;
};
```

## 실수 포인트

- 전체 목록 snapshot으로 rollback하면 그 사이 성공한 다른 요청까지 덮어쓸 수 있다.
- optimistic state만 남기고 서버 응답을 버리면 id, 정렬, 권한 결과가 어긋난다.
- 같은 항목의 중복 요청을 허용하면 빠른 연속 클릭으로 상태가 꼬인다.
- temporary id를 만들지 않으면 성공 응답을 어느 optimistic state와 맞출지 불명확해진다.
