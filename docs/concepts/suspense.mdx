# Suspense

## API reference

- `<Suspense fallback={...}>`: 자식이 아직 렌더링 준비를 끝내지 못하면 `fallback`을 보여주고, 준비되면 실제 UI로 바꾼다.
- `lazy(() => import(...))`: 컴포넌트 코드를 동적으로 불러온다. 로드가 끝나기 전에는 가까운 Suspense boundary가 pending 상태를 맡는다.
- thrown Promise/resource: 렌더링 중 읽은 값이 아직 준비되지 않았다는 신호다. Promise나 resource는 렌더링 바깥 계층에서 안정적으로 만들어야 한다.
- Error Boundary: lazy import 실패, rejected Promise, 렌더링 에러를 처리한다. Suspense는 pending만 처리한다.

## 쓰는 곳

- route나 패널처럼 사용자가 기다리는 단위의 loading boundary를 만들 때 쓴다.
- `React.lazy`로 code splitting한 컴포넌트를 감쌀 때 쓴다.
- framework, router loader, cache처럼 Suspense를 지원하는 async resource를 읽을 때 쓴다.

## 바로 쓰는 코드

```tsx title="SettingsRoute.tsx"
import { lazy, Suspense } from "react";

const SettingsPanel = lazy(() => import("./SettingsPanel"));

const SettingsLoading = () => (
  <section aria-busy="true">
    <h2>설정을 불러오는 중입니다.</h2>
  </section>
);

const SettingsError = () => (
  <section role="alert">
    <h2>설정을 불러오지 못했습니다.</h2>
  </section>
);

const SettingsRoute = () => (
  <ErrorBoundary fallback={<SettingsError />}>
    <Suspense fallback={<SettingsLoading />}>
      <SettingsPanel />
    </Suspense>
  </ErrorBoundary>
);
```

## 실수 포인트

- Suspense fallback과 Error Boundary fallback을 같은 UI로 만들면 pending과 실패를 구분하기 어렵다.
- 렌더링 중 `fetch()`를 새로 만들면 매 렌더링마다 다시 suspend될 수 있다.
- boundary를 너무 넓게 잡으면 작은 지연도 화면 전체를 fallback으로 바꾼다.
- fallback과 실제 콘텐츠 크기가 크게 다르면 layout shift가 커진다.
