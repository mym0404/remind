# 컴포넌트 분리

## API reference

컴포넌트를 나누면 파일보다 API가 먼저 생긴다. 바깥에서 넘길 값과 이벤트를 props 타입으로 고정하고, 내부 state 규칙은 소유한 컴포넌트 안에 둔다.

```tsx title="props-contract.tsx"
type Todo = {
  id: string;
  title: string;
  done: boolean;
};

type TodoListProps = {
  todos: Todo[];
  children?: React.ReactNode;
  onToggle: (id: Todo["id"]) => void;
};

export const TodoList = ({ todos, children, onToggle }: TodoListProps) => (
  <section>
    {children}
    <ul>
      {todos.map((todo) => (
        <li key={todo.id}>
          <button type="button" onClick={() => onToggle(todo.id)}>
            {todo.done ? "완료" : "진행 중"}
          </button>
          {todo.title}
        </li>
      ))}
    </ul>
  </section>
);
```

- Props type declaration은 컴포넌트의 입력 계약이다.
- `children`은 컴포넌트 사이에 끼워 넣을 UI가 있을 때만 연다.
- callback props는 내부 함수 이름이 아니라 바깥에서 일어난 이벤트 이름으로 둔다.
- named export는 조립하는 쪽에서 필요한 컴포넌트만 명시해서 가져오게 한다.

Controlled props는 값과 변경 이벤트를 한 세트로 둔다. 초깃값만 받는 컴포넌트와 섞지 않는다.

```tsx title="controlled-filter.tsx"
type FilterValue = "all" | "active" | "done";

type TodoFilterTabsProps = {
  value: FilterValue;
  onValueChange: (nextValue: FilterValue) => void;
};

export const TodoFilterTabs = ({
  value,
  onValueChange,
}: TodoFilterTabsProps) => (
  <div role="tablist" aria-label="할 일 필터">
    {(["all", "active", "done"] as const).map((filter) => (
      <button
        key={filter}
        type="button"
        aria-selected={value === filter}
        onClick={() => onValueChange(filter)}
      >
        {filter}
      </button>
    ))}
  </div>
);
```

Composition boundary는 상위 컴포넌트가 조립을 맡고, 하위 컴포넌트가 받은 props 안에서만 그리는 선이다. 하위 컴포넌트가 부모의 state 전환 규칙을 알기 시작하면 경계가 흐려진다.

## 언제 쓰나

state 전환 규칙과 JSX 흐름이 서로 가리면 역할을 나눈다.

- JSX가 길어져 화면 흐름을 따라가기 어렵다.
- 같은 UI 조각이 두 번 이상 나오거나 이름 붙일 만큼 의미가 있다.
- state 로직이 UI보다 더 많은 줄을 차지한다.
- 여러 형제 컴포넌트가 같은 state를 함께 쓴다.
- 멀리 떨어진 컴포넌트가 같은 값을 읽거나 바꾼다.

state는 사용하는 곳에 최대한 가깝게 둔다. 여러 형제가 공유하면 공통 부모로 올리고, 여러 단계 아래에서 계속 쓰면 Context를 검토한다.

## 바로 쓰는 코드

```tsx title="useTodoList.ts"
// ...

const useTodoList = () => {
  const [todos, setTodos] = useState(initialTodos);
  const [filter, setFilter] = useState("all");

  const visibleTodos = useMemo(() => {
    // ...
  }, [filter, todos]);

  // ...

  // Keep state transitions inside the owner.
  const toggleTodo = (id) => {
    setTodos((currentTodos) =>
      currentTodos.map((todo) =>
        todo.id === id ? { ...todo, done: !todo.done } : todo, // [!code highlight]
      ),
    );
  };

  return {
    filter,
    visibleTodos,
    setFilter,
    toggleTodo,
  };
};
```

```tsx title="TodoPage.tsx"
// ...

const TodoPage = () => {
  const { filter, visibleTodos, setFilter, toggleTodo } = useTodoList(); // [!code highlight]

  return (
    <section>
      <h2>할 일</h2>
      {/* Children receive values and events, not state rules. */}
      <TodoFilterTabs value={filter} onChange={setFilter} />
      <TodoList todos={visibleTodos} onToggle={toggleTodo} />
    </section>
  );
};

const TodoFilterTabs = ({
  value,
  onChange,
}) => <div>{/* ... */}</div>;

const TodoList = ({
  todos,
  onToggle,
}) => {
  if (todos.length === 0) {
    return <p>표시할 일이 없습니다.</p>;
  }

  return (
    <ul>
      {todos.map((todo) => (
        <TodoItem
          key={todo.id} // [!code highlight]
          todo={todo}
          onToggle={onToggle}
        />
      ))}
    </ul>
  );
};

const TodoItem = ({
  todo,
  onToggle,
}) => <li>{/* ... */}</li>;
```

`TodoPage`는 컴포넌트를 조립한다. `useTodoList`는 state와 변경 함수를 가진다. `TodoFilterTabs`, `TodoList`, `TodoItem`은 받은 props로만 화면을 그린다.

Props 이름은 바깥에서 보는 동작을 기준으로 정한다. Controlled API는 `open`, `defaultOpen`, `onOpenChange`처럼 세트를 맞춘다.

## 실수 포인트

- 컴포넌트가 작다는 이유만으로 파일부터 나누지 않는다.
- Context에는 멀리 떨어진 여러 컴포넌트가 함께 쓰는 값만 둔다.
- callback props는 `onSelect`, `onSubmit`, `onClose`, `onOpenChange`처럼 외부 이벤트 이름으로 만든다.
- 반복 렌더링의 `key`에는 안정적인 id를 쓴다.
- 내부 구현에 맞춘 props 이름은 피한다.
