# Error Boundary

## API reference

- `static getDerivedStateFromError(error)`: 자식 렌더링 에러 뒤 다음 render에서 쓸 fallback state를 반환한다. 로그 전송 같은 side effect는 넣지 않는다.
- `componentDidCatch(error, info)`: commit 이후 호출된다. `info.componentStack`과 함께 에러 리포팅을 보낼 위치다.
- fallback state: boundary 내부에 `hasError` 같은 값을 두고, 실패 후 `children` 대신 fallback을 렌더링한다.
- reset by key: 실패한 boundary의 state는 자동으로 돌아오지 않는다. 부모가 `key`를 바꾸면 boundary가 remount되어 초기 state로 시작한다.
- 못 잡는 에러: 이벤트 핸들러, async callback, Promise rejection, server-side rendering 에러, boundary 자신이 던진 에러는 직접 잡지 못한다.

## 쓰는 곳

route, page, 큰 panel처럼 한 영역의 렌더 실패가 앱 전체를 멈추면 안 되는 곳에 둔다. 너무 바깥에 하나만 두면 fallback이 커지고, 너무 잘게 두면 retry와 로그가 흩어진다.

## 바로 쓰는 코드

```tsx title="ErrorBoundary.tsx"
type ErrorBoundaryProps = {
  fallback: React.ReactNode;
  onError?: (error: unknown, info: React.ErrorInfo) => void;
  children: React.ReactNode;
};

type ErrorBoundaryState = {
  hasError: boolean;
};

class ErrorBoundary extends React.Component<
  ErrorBoundaryProps,
  ErrorBoundaryState
> {
  state: ErrorBoundaryState = { hasError: false };

  static getDerivedStateFromError(): ErrorBoundaryState {
    return { hasError: true }; // [!code highlight]
  }

  componentDidCatch(error: unknown, info: React.ErrorInfo) {
    this.props.onError?.(error, info); // [!code highlight]
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback; // [!code highlight]
    }

    return this.props.children;
  }
}
```

retry 버튼은 fallback 안에 두되, boundary가 자기 state를 직접 되돌리게 만들기보다 부모에서 `key`를 바꿔 remount한다.

## 실수 포인트

- 이벤트 핸들러와 `fetch` 실패는 호출한 코드에서 직접 처리한다.
- fallback 컴포넌트가 같은 에러를 다시 던지지 않게 둔다.
- Suspense fallback은 로딩 UI이고, Error Boundary fallback은 실패 UI다.
- retry가 필요하면 실패한 자식 데이터와 boundary `key`가 같이 새로 시작되는지 확인한다.
