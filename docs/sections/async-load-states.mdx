# Async Load States

## 문제

네트워크 요청의 loading, error, data, empty 상태를 분리한다.

## 핵심 개념

- 상태는 `loading | success | empty | error`로 나눈다.
- 빈 배열은 success 목록이 아니라 empty state다.
- 실패는 loading을 끝내고 error UI로 보인다.

## 풀이 흐름

`useEffect`에서 loading으로 시작하고 loader 결과를 상태 union으로 바꾼다. cleanup flag로 unmount 뒤 `setState`를 막는다.

## 핵심 코드

```tsx
type LoadState<T> =
  | { status: "loading" }
  | { status: "success"; data: T[] }
  | { status: "empty" }
  | { status: "error"; message: string };

useEffect(() => {
  let cancelled = false;
  setState({ status: "loading" });
  fetchProducts()
    .then((data) => setState(data.length === 0 ? { status: "empty" } : { status: "success", data }))
    .catch(() => !cancelled && setState({ status: "error", message: "상품을 불러오지 못했습니다." }));
  return () => { cancelled = true; };
}, []);
```

## 테스트가 보는 것

- mount 직후 loading이 보이고, 요청 후 상품이 나타나는지 본다.
- 빈 응답과 실패 응답을 data state와 분리해 검증한다.

## 실수 포인트

- 빈 배열을 success로만 처리하면 화면에 아무것도 없어 원인을 알기 어렵다.
- catch에서 상태를 바꾸지 않으면 실패가 영원한 loading처럼 보인다.
